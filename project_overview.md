# ContextEngineAI Project Overview

### 🔬 Project Overview & Aims
The **ContextEngineAI** project is a comprehensive and intelligent multi-source Artificial Intelligence research assistant. Its primary aim is to take user research queries and provide deeply grounded, accurate answers heavily reinforced by context and citations rather than just relying on an LLM's hallucinated knowledge. 

It achieves this by combining a traditional RAG (Retrieval-Augmented Generation) pipeline with dynamic search capabilities. It choreographs multiple "Agents" using **CrewAI Flows** to run parallel tasks: fetching document knowledge, pulling in conversation history, scraping recent web data, and pulling down academic papers. Later, it logically reviews all retrieved context, filters out irrelevant noise, and synthesizes a final, highly coherent response securely backed by structured citations.

---

### 🚀 Things That Are Implemented
Currently, the system is actively running a robust multi-agent pipeline and User Interface. The active implementations include:
*   **Vector RAG Pipeline:** Takes uploaded PDF documents, parses them structurally into chunks, maps their semantic meaning into embeddings, and stores them in a local Milvus vector database for incredibly fast similarity search.
*   **Memory Layer Framework:** A system to persist session history, save past questions from users, and summarize assistant answers to prevent token limits from maxing out while referencing older messages contextually.
*   **Streamlit Web Interface:** A graphical end-user app (`app.py`) allowing a user to upload PDFs, watch real-time processing indicators, issue chat-based questions, and view rich markdown answers that feature interactive citation dropdowns for sources.
*   **Multi-Agent Ecosystem:** Contains several distinct LLM agents defined in your CrewAI setup that work in harmony:
    *   **RAG Agent**: Extracts context from the DB corresponding to user queries.
    *   **Memory Agent**: Dips into long-term history to remember context boundaries.
    *   **Web Search Agent**: Triggers live web searches to patch gaps not found in local documents.
    *   **ArXiv Academic Agent**: Used for pulling and extracting from scholarly papers.
    *   **Evaluator Agent**: Grades the relevancy and confidence of the data generated by the first four agents.
    *   **Synthesizer Agent**: The final writer who blends everything into the final response block.

### File Names and Their Purposes
*   **`app.py`**: The main frontend stream interface for user interactions. It manages chat history UI, file uploads, and instantiates the workflow payload logic.
*   **`README.md`**: The centralized documentation highlighting project features, diagrams, and setups.
*   **`milvus_lite.db`**: A local database backend storing the document vector embeddings for fast lookup.
*   **`src/workflows/flow.py`**: The central controller of the project coordinating logic between defining State, kicking off the RAG/Memory/Web tasks, triggering Evaluations, and finally ending on the response Generator. 
*   **`src/workflows/agents.py`**: The Factory class that instantiates all individual CrewAI agents. 
*   **`src/workflows/tasks.py`**: Detailed instruction payloads mapping tasks against agents (like what the specific RAG task asks of the RAG agent).
*   **`src/tools/rag_tool.py`**: The strict CrewAI tool equipping the RAG agent with DB contexts matching user queries without forcing generation.
*   **`src/tools/memory_tool.py`**: The tool linking the conversational context agent with Zep Cloud.
*   **`src/tools/web_search_tool.py`**: Exposes Firecrawl search mechanics directly to the Web Search agent.
*   **`src/tools/arxiv_tool.py`**: Enables the academic tracking agent to interact directly with Arxiv APIs.
*   **`src/rag/rag_pipeline.py`**: Acts as a massive wrapper class containing document parsers, embeddings, database connectors, and generators into a singular RAG workflow object.
*   **`src/rag/retriever.py`**: Handles creating/inserting/searching the Milvus Lite DB instance schema.
*   **`src/rag/embeddings.py`**: Interacts with the Voyage models to convert chunks of strings into mathematical vector lists.
*   **`src/document_processing/doc_parser.py`**: Wraps the TensorLake integration to intelligently analyze layout layouts of PDFs instead of arbitrarily cutting lines in half.
*   **`src/memory/memory.py`**: Manages storing, loading, and summarizing thread-based conversation payloads onto the Zep Cloud.
*   **`config/agents/research_agents.yaml`**: Configuration file containing the specific `Role`, `Goal`, and `Backstory` prompts mapped to each active agent.
*   **`config/tasks/research_tasks.yaml`**: The configuration mapped to the expected output instructions corresponding to each active agent's task.

### 🔑 What Each API Key Does
*   **TENSORLAKE (`TENSORLAKE_API_KEY`)**: Deep Document Parsing. Extracts structured textual chunks carefully out of messy PDFs.
*   **VOYAGE (`VOYAGE_API_KEY`)**: Semantic Embeddings. Calculates vector associations from text chunks to run similarity searches.
*   **ZEP (`ZEP_API_KEY`)**: The Long-Term Memory Cloud. Manages, summaries, and retrieves context from the conversational state mapping.
*   **OPENAI (`OPENAI_API_KEY`)**: The Cognitive Engine. Powers all CrewAI agents, allowing them to think, choose, and write.
*   **FIRECRAWL (`FIRECRAWL_API_KEY`)**: Remote Web crawling. Searches the live internet to pull missing contextual answers actively.
